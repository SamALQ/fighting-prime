## Project Summary

Fighting Prime Academy is a premium online Muay Thai training platform built in partnership with ONE Championship athlete Jake Peacock. It delivers structured video courses, technique breakdowns, and a gamified progress system through a Next.js web application.

**Production domain:** `fightingprime.com`

**Current state:** The app is a **frontend-only prototype**. The UI, course structure, video playback, gamification logic, and design system are all built and functional — but there is **no backend, no database, no real auth, and no payment processing**. All data is hardcoded in TypeScript files under `data/`. All user state (auth, progress, watch time) lives in `localStorage`. There is a single hardcoded test user.

**Goal:** Transform this into a fully production-ready platform with real authentication, a Supabase database, Stripe billing, secure video delivery, an admin CMS for content uploads, and the Fighter Elite premium feature — then launch it to replace the current Webflow + Memberstack + Make system.

## Architecture Decisions

These decisions have already been made. Follow them consistently:

1. **Supabase is the backend.** Use Supabase for auth, database (Postgres), and row-level security. Do not introduce a separate backend server or ORM. Use the Supabase JS client (`@supabase/supabase-js`) and the SSR helpers (`@supabase/ssr`) for Next.js App Router.

2. **Stripe connects directly.** No Memberstack, no payment middleware. Stripe Checkout for purchases, Stripe Customer Portal for self-service billing, Stripe Webhooks to sync subscription state to the `subscriptions` table in Supabase.

3. **S3 stays for video storage.** Videos are already hosted on AWS S3. Do not migrate them. Instead, serve them through presigned URLs generated server-side after verifying the user's subscription status. Free episodes bypass the subscription check.

4. **Admin uploads go to S3 via presigned URLs.** The browser uploads directly to S3 using a presigned upload URL generated by an API route. The server never handles the raw video file. This keeps things fast and avoids Vercel's request size limits.

5. **Role-based access via `profiles.role`.** Three roles: `user` (default), `instructor` (Jake), `admin` (Sam). The `profiles` table extends Supabase's `auth.users`. RLS policies enforce access. Admin routes are protected server-side.

6. **Progressive migration.** Replace things one system at a time. The checklist is ordered so that each phase produces a working app — you should be able to deploy after each phase without breaking what came before.

## Key Business Context

- There are currently **zero active monthly subscribers**.
- There is **one active yearly subscriber** — their Stripe customer ID needs to be mapped to a new Supabase user account during migration. Do not disrupt their billing.
- The current Webflow + Memberstack + Make system is being fully replaced. Once the new app is live, the old system gets shut down.
- Jake Peacock (the instructor) is not technical. The admin upload system must be extremely simple — large buttons, clear labels, drag-and-drop, minimal steps. Think "upload a video to YouTube" level of simplicity.
- **Fighter Elite** is a premium tier feature where subscribers can upload monthly sparring/technique footage and receive a personalized video breakdown from Jake. This requires a two-way upload workflow with status tracking.

## Known Issues to Fix During Build

These are problems identified in the current codebase that must be resolved as part of the migration:

### Critical Security
- **Hardcoded credentials** — `sam@alqdigital.com` / `S` are in the client bundle via `useAuth`. Remove entirely when Supabase Auth is in place.
- **Exposed video URLs** — S3 URLs are hardcoded in `data/episodes.ts` and visible in the frontend bundle. Anyone can extract them. Fix by serving through presigned URLs.
- **No server-side auth** — All auth checks are client-side `localStorage` reads. A user can manually set `isLoggedIn: "true"` in their browser and access everything. Fix with Supabase session middleware.

### Broken/Placeholder Features
- **Fighter Elite upload** — Drag-and-drop UI exists but does nothing. Wire to real S3 uploads + `elite_submissions` table.
- **Comments on breakdowns** — UI-only placeholder. Either implement or remove for launch.
- **Assignments** — Referenced in episode data but not implemented. Either implement or remove the UI references.
- **Achievements** — Grid exists with hardcoded mock data. Replace with real achievement logic or clearly mark as "coming soon."
- **Recent activity feed** — Uses mock data. Replace with real queries from `user_progress`.

### UX Gaps
- **No mobile navigation** — NavBar links overflow on small screens. Build a hamburger/slide-out menu.
- **No error boundaries** — A crash in VideoPlayer or Dashboard white-screens the whole app.
- **No loading states** — Pages that will now fetch async data need skeleton loaders.
- **Sandbox page** — `/home-sandbox1` is an experiment. Remove before launch.

## How to Use the Checklist

The file `FPA_Launch_Checklist.md` contains every task organized into 8 phases. Here's how to work through it:

### Rules

1. **Work phase by phase, in order.** Phase 2 depends on Phase 1. Phase 3 depends on Phase 2. Don't skip ahead.

2. **Within a phase, work section by section.** Each section (e.g., 1.1, 1.2, 1.3) is a logical unit. Complete one before starting the next.

3. **Each checkbox is a concrete deliverable.** When you complete a task, the app should still build and run. Don't leave things half-wired.

4. **Test after every section.** After completing section 1.2 (schema), verify the tables exist and seed data loads. After 1.3 (auth), verify signup/login/logout works end-to-end. Don't accumulate untested work.

5. **Keep the existing UI working.** The frontend design is already done and should not be redesigned during this process. You're replacing the data layer underneath it, not rebuilding the interface. If a component currently reads from `data/courses.ts`, update it to read from Supabase — but keep the same visual output.

6. **When in doubt, ask.** If a task is ambiguous or requires a decision not covered here (e.g., "should we implement comments for launch or defer?"), ask rather than assuming.

### Phase Summary

| Phase | Focus | Key Deliverable |
|---|---|---|
| 1 | Supabase + Auth | Real database, real user accounts, role-based access |
| 2 | Stripe | Working payments, subscription sync, content gating |
| 3 | Data Migration | All localStorage replaced with Supabase, cross-device sync |
| 4 | Video Infrastructure | Secure video delivery, admin upload CMS for Jake |
| 5 | Fighter Elite | Full submission → review → response workflow |
| 6 | Security & Code Quality | Remove all vulnerabilities, error boundaries, mobile nav |
